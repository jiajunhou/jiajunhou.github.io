RabbitMQ在订单业务中的使用

一、引言

注：这篇文章适用于已经了解消息中间件相关知识的，其实理论并不复杂，尽管文章内也会对一些名词进行介绍和分析，但还是建议没有了解过的可以先移步百度学习一下，还是比较容易的！

在含有交易业务系统里，订单是非常关键的。随着业务增长，订单系统面临高并发、系统耦合等挑战。RabbitMQ 作为强大的消息队列，能有效解决这些问题。本文将详细介绍如何用 Java 结合 RabbitMQ 优化订单业务。

二、基本介绍

1. 什么是 RabbitMQ？

RabbitMQ 是遵循 AMQP 协议的开源消息代理，它由生产者、交换机、队列和消费者构成。生产者发送消息到交换机，交换机按规则把消息路由到队列，消费者从队列获取消息处理。

2. 为何选 RabbitMQ？

  • 异步处理：能把订单业务流程异步化，提高系统响应速度。
  • 系统解耦：降低订单系统与其他系统（如库存、支付）的耦合度。
  • 可靠性强：支持消息持久化、确认机制，保证消息不丢失。

2. 分析使用场景

RabbitMQ 根据不同的生产者 - 消费者模型分为六种模式。

简单模式

优点
  • 实现简单：代码结构和逻辑都非常基础，对于初学者或者简单业务场景来说，易于理解和实现。开发成本低，能快速搭建起订单消息传递机制。
  • 低延迟：由于消息直接从生产者到消费者，没有复杂的中间环节，消息传递的延迟较低，适合对实时性要求较高的简单订单通知场景。
缺点
  • 一致性问题
    • 数据一致性难以保证：在简单模式下，若消费者处理消息失败，没有完善的重试和补偿机制，容易导致订单数据不一致。例如，订单创建消息发送后，消费者处理时因数据库异常未能成功创建订单记录，但生产者无法得知，就会造成订单数据在生产者和消费者两端不一致。
  • 扩展性与可靠性问题
    • 缺乏扩展性：只能有一个消费者处理消息，无法应对高并发订单量。当业务增长，订单量增加时，系统处理能力会成为瓶颈。
    • 可靠性低：没有消息确认和持久化机制的默认支持，一旦消息在传输过程中丢失或者消费者出现故障，消息就会丢失，影响订单业务的正常进行。
典型案例

在小型电商系统中，新订单创建后，需要将订单信息传递给后台的订单处理服务进行初步的订单审核和记录。由于业务流程简单，只需一个消费者处理订单消息，所以适合使用简单模式。

工作队列模式

优点
  • 并发处理能力强：多个消费者共同处理订单消息，能有效提高订单处理的并发能力，应对高并发订单场景，如电商大促时的订单处理。
  • 负载均衡：RabbitMQ 会自动将消息均匀分配给各个消费者，避免单个消费者负载过高，提高了系统的整体性能。
缺点
  • 一致性问题
    • 消息顺序性难以保证：多个消费者并行处理消息，无法保证消息的处理顺序与发送顺序一致。在一些对订单处理顺序有严格要求的业务场景中，如订单的创建、支付、发货流程，可能会导致业务逻辑混乱。
    • 部分处理失败导致的不一致：如果某个消费者处理消息失败，可能会导致部分订单处理不完整，而其他消费者继续处理后续消息，造成订单数据的不一致。例如，一个订单的库存扣减失败，但后续的订单状态更新等操作已经完成，就会出现数据不一致的情况。
  • 管理与协调问题
    • 消费者管理复杂：随着消费者数量的增加，对消费者的管理和监控变得复杂。需要确保每个消费者的正常运行，并且在消费者出现故障时能够及时处理。
    • 消息竞争问题：多个消费者竞争处理消息，可能会出现资源竞争的情况，如多个消费者同时访问数据库，增加了数据库的并发压力，甚至可能导致数据冲突。
典型案例

在电商大促期间，订单量会急剧增加。为了提高订单处理的并发能力，使用多个订单处理服务实例（消费者）共同监听一个订单队列，每个消费者负责处理一部分订单消息。

发布 / 订阅模式

优点
  • 系统解耦：生产者和多个消费者之间通过交换机进行消息传递，实现了系统的解耦。各个系统可以独立开发、部署和维护，提高了系统的可维护性和扩展性。
  • 消息广播：能够将订单消息同时发送给多个相关系统，适用于订单状态变更通知等需要多系统协同的业务场景。
缺点
  • 一致性问题
    • 多系统处理结果不一致：不同的消费者处理消息的速度和结果可能不同，容易导致多系统之间的数据不一致。例如，订单状态变更消息广播后，物流系统和客服系统处理结果不一致，可能会给用户带来困惑。
    • 消息丢失或重复消费：在广播过程中，可能会出现消息丢失的情况，或者由于消费者故障等原因导致消息重复消费，影响订单数据的一致性。
  • 性能与资源问题
    • 性能开销大：交换机需要将消息广播到所有绑定的队列，随着队列数量的增加，会带来较大的性能开销。
    • 资源占用高：每个消费者都需要维护一个队列，会占用较多的系统资源。
典型案例

当订单状态发生变化（如已支付、已发货等）时，需要通知多个相关系统，如物流系统、客服系统、营销系统等。使用发布 / 订阅模式，订单系统将订单状态变更消息发送到交换机，各个相关系统通过绑定的队列获取消息并进行相应的业务操作。

路由模式

优点
  • 精准消息路由：根据路由键将消息精准地发送到指定的队列，实现了消息的精准投递，适用于不同类型订单的分类处理，如退款订单、普通订单等。
  • 业务逻辑清晰：通过路由键可以明确区分不同的业务场景，使业务逻辑更加清晰，便于开发和维护。
缺点
  • 一致性问题
    • 路由键配置错误导致的不一致：如果路由键配置错误，消息可能会被错误地发送到其他队列，导致订单处理逻辑混乱，数据不一致。
    • 部分队列处理失败的影响：当某个队列的消费者处理消息失败时，可能会影响整个业务流程的一致性。例如，退款订单的某个处理队列出现问题，可能会导致退款流程无法正常完成。
  • 灵活性与可维护性问题
    • 路由键管理复杂：随着业务的发展，路由键的数量和规则会不断增加，管理和维护路由键变得复杂，容易出错。
    • 缺乏灵活性：路由规则相对固定，当业务规则发生变化时，需要修改路由键的配置，不够灵活。
典型案例

在订单退款业务中，不同的退款原因（如商品质量问题、用户主观原因等）需要不同的处理流程。使用路由模式，订单系统根据退款原因生成相应的路由键，并将退款消息发送到交换机，不同的退款处理逻辑对应的消费者通过绑定特定的路由键来接收相关的退款消息。

主题模式

优点
  • 灵活的消息路由：使用通配符进行消息路由，支持更复杂的消息筛选和匹配规则，能够根据订单的不同属性（如订单时间、订单金额等）灵活地将消息路由到不同的队列。
  • 动态适应业务变化：当业务规则发生变化时，只需修改消费者绑定的路由键，而不需要修改生产者的消息发送逻辑，提高了系统的灵活性和可维护性。
缺点
  • 一致性问题
    • 通配符匹配的不确定性：通配符匹配可能会导致消息被误匹配到不相关的队列，影响订单数据的一致性。例如，使用 orders.* 匹配订单消息时，可能会将一些不相关的消息也匹配进来。
    • 多队列处理结果不一致：由于消息可以被路由到多个队列，不同队列的消费者处理结果可能不一致，增加了数据一致性管理的难度。
  • 性能与理解成本问题
    • 性能开销较大：通配符匹配需要进行更复杂的字符串匹配操作，相比于精确匹配，会带来一定的性能开销。
    • 理解成本高：对于开发人员来说，理解和配置复杂的通配符规则需要一定的时间和精力，增加了开发和维护的成本。
典型案例

在订单数据分析场景中，订单系统根据不同的维度（如订单时间、订单金额、商品类别等）生成包含相应信息的路由键，并将订单数据消息发送到交换机。数据分析服务通过绑定不同的主题模式路由键（如 orders.# 获取所有订单相关数据，orders.monthly.# 获取每月订单数据等），来获取所需的订单数据进行分析。

RPC 模式

优点
  • 同步交互方便：允许客户端发送请求消息到服务器，并等待服务器返回响应消息，适用于需要同步交互的业务场景，如订单支付结果查询。
  • 远程调用简单：通过消息队列实现远程调用，简化了分布式系统中不同服务之间的通信过程。
缺点
  • 一致性问题
    • 网络延迟和故障导致的不一致：由于是同步调用，网络延迟或服务器故障可能会导致客户端长时间等待响应，甚至出现请求超时的情况。在这种情况下，客户端和服务器之间的数据可能会不一致，例如客户端以为请求失败，但服务器实际上已经处理了请求。
    • 事务一致性难以保证：在涉及多个服务的 RPC 调用中，保证事务的一致性比较困难。如果某个服务调用失败，可能会导致部分业务操作已经完成，而其他部分未完成，造成数据不一致。
  • 性能与扩展性问题
    • 性能瓶颈：同步调用方式会使客户端线程长时间阻塞，影响系统的并发处理能力。在高并发场景下，容易成为系统的性能瓶颈。
    • 扩展性有限：随着业务的发展，RPC 调用的数量和复杂度会增加，系统的扩展性会受到限制。例如，增加新的服务或修改服务接口可能会影响到其他相关服务的调用。
典型案例

在订单支付环节，订单系统需要调用第三方支付接口获取支付结果。使用 RPC 模式，订单系统作为客户端发送包含支付订单号等信息的请求消息到 RabbitMQ 的 RPC 队列，第三方支付接口服务作为服务器接收请求并处理，然后将支付结果通过另一个队列返回给订单系统。

四、存在的问题

生产者消息可靠性投递

消息投递路径：

producer–>rabbitmq broker –> exchange –>queue –> consumer

消息可靠性投递：如何保证生产者发送的消息不会出现丢失或者投递失败。

在使用 RabbitMQ 的时候，作为 消息发送方 希望杜绝任何消息丢失或投递失败场景。RabbitMQ 为我们提供了两种方式用来控制消息的投递可靠性模式。

  • confirm 确认模式：消息从 producer 到 exchange，则会返回一个confirmCallback。不管消息是否成功到达exchange，confirmCallback 都会执行，通过回调函数的参数，可以判断消息传递是否成功。
  • return 退回模式：消息从 exchange 到 queue，投递失败则会返回一个 returnCallback。

我们将利用这两个 Callback 控制消息的可靠性投递

Consumer ACK

ack 指 Acknowledge，确认。表示消费端收到消息的确认方式。

有三种确认方式：

  • 自动确认（默认）：acknowledge-mode=none
  • 手动确认：acknowledge-mode=manual
  • 根据异常情况确认：acknowledge-mode=auto（这种方式使用麻烦，不推荐）

其中自动确认是指，当消息一旦被 Consumer 接收到，则自动确认收到，并将相应 message 从RabbitMQ 的消息缓存中移除。但是在实际业务处理中，很可能消息接收到，业务处理出现异常，那么该消息就会丢失。

如果设置了手动确认方式，则需要在业务处理成功后，调用 channel.basicAck() ，手动签收，如果出现异常，则调用 channel.basicNack() 方法，让其自动重新发送消息。

消费端限流

假设一个场景，首先，我们 Rabbitmq 服务器积压了有上万条未处理的消息，我们随便打开一个消费者客户端，会出现这样情况：巨量的消息瞬间全部推送过来，但是我们单个客户端无法同时处理这么多数据。

当数据量特别大的时候，我们对生产端限流肯定是不科学的，因为有时候并发量就是特别大，有时候并发量又特别少，我们无法约束生产端，这是用户的行为。所以我们应该对消费端限流，用于保持消费端的稳定，当消息数量激增的时候很有可能造成资源耗尽，以及影响服务的性能，导致系统的卡顿甚至直接崩溃。

过期时间TTL

TTL 全称 Time To Live （存活时间/过期时间）。

过期时间TTL表示可以对消息设置预期的时间，在这个时间内都可以被消费者接收获取；当消息到达存活时间后，还没有被消费，会被自动清除。RabbitMQ可以对消息和队列设置TTL。

  • 对整个队列（Queue）设置过期时间：通过队列属性设置，队列中所有消息都有相同的过期时间。（推荐）

  • 对消息设置过期时间：对消息进行单独设置，每条消息TTL可以不同。

如果上述两种方法同时使用，则消息的过期时间以两者之间TTL较小的那个数值为准。消息在队列的生存时间一旦超过设置的TTL值，就称为dead message (死信)，消息被丢弃。

死信队列

DLX，全称为 Dead Letter Exchange , 可以称之为死信交换机。当消息在一个队列中变成死信（Deadmessage）之后，它可以被重新发送到另一个交换机中，这个交换机就是DLX ，绑定DLX的队列就称之为死信队列。

DLX也是一个正常的交换机，和一般的交换机没有区别，它能在任何的队列上被指定，实际上就是设置某一个队列的属性。当这个队列中存在死信时，Rabbitmq就会自动地将这个消息重新发布到设置的DLX上去，进而被路由到另一个队列，即死信队列。

消息成为死信的三种情况：

  1. 队列消息长度到达限制

  2. 消费者拒绝消费消息（basicNack/basicReject），并且不把消息重新放到原目标队列（requeue=false）

  3. 原队列存在消息过期设置，消息到达超时时间未被消费

延迟队列

延迟队列，即消息进入队列后不会立即被消费，只有到达指定时间后，才会被消费。

需求：

  1. 下单后，30分钟未支付，取消订单，回滚库存

  2. 新用户注册成功7天后，发送短信问候

实现方式：

  1. 定时器：定义时间间隔，循环执行指定任务；

    使用定时器完成需求1，下单之后，会向订单表中插入一条数据，该数据包含下单时间，通过定时器 + 定时任务，从订单表中读取订单数据和当前时间比对，如果时间间隔超过30分钟，判断订单状态是否支付，如果未支付，则取消订单，回滚库存。

Producer 将业务数据保存到数据库（①）

Producer 将消息发送给Q1（②），Consumer 监听 Q1（④），拿到消息消费，Consumer 操作数据库，这就是消息发送正常的情况（Producer -> MQ -> Consumer）

  • 正常情况：Consumer 作为发送方，发送确认消息给Q2（⑤），回调检查服务监听Q2（⑥），收到确认消息之后，将消息写入消息数据库（⑦）

  • 失败情况：Producer 将业务数据保存到数据库成功了，但是发送消息失败（②）了，那么Consumer肯定收不到消息

    • 解决方案：Producer 发送消息之后，在指定时间之后，再次发送消息（③延迟发送消息），② 和 ③ 发送的消息是一样的，但是一个发送给Q1，一个发送给Q3，回调检查服务监听Q3（⑧），回调检查服务比对 Q3 里面的消息 和 MDB 数据库的消息是否一致
      • 如果一致，那么说明这个消息肯定被消费过了，也就是正常情况
      • 如果不一致，那么说明这个消息肯定没有被消费过，那么也就是说发送消息失败，回调检查服务通过远程调用方式，调用Producer，重新发送消息
  • 特殊情况：发送消息失败、延迟发送消息也失败了
    • 解决方案：定时检查服务，每隔指定时间，检查 DB 和 MDB 数据是否匹配，如果匹配失败，定时检查服务通过远程调用方式，调用Producer，重新发送消息（⑨）

消息幂等性处理

幂等性指一次和多次请求某一个资源，对于资源本身应该具有同样的结果。也就是说，其任意多次执行对资源本身所产生的影响均与一次执行的影响相同。

在MQ中指，消费多条相同的消息，得到与消费该消息一次相同的结果。

  • 举例说明：张三购买了一件商品，花费了500元。Producer 发送一条消息给MQ，Consumer 从队列中获取数据进行消费，假设Consumer出现了问题，那么会导致消息重发，此时MQ中就存在了两条相同的数据，如果Consumer后续正常了，肯定要保证只会扣款500，而不是1000。
